# Refious Security — Audit Methodology

High-level overview of how Refious approaches a smart contract security engagement. This document describes **what** the methodology delivers and **how** an engagement is structured, not the internal tooling that underpins it.

---

## Engagement Phases

### 1. Scoping & Threat Model

**Inputs:** target codebase, commit hash(es), architecture documentation, intended deployment context, trust assumptions.

**Activities:**
- Define exact scope: contract files in, dependencies out, treatment of upgradeable proxies.
- Identify privileged roles and trust boundaries.
- Build the threat model — who can do what, to whom, under which assumptions.
- Map external dependencies (oracles, bridges, third-party protocols) and document trust assumptions on each.
- Agree on out-of-scope boundary in writing.

**Deliverable:** scope document (commit hashes + file list + threat model + assumptions).

---

### 2. Architecture Review

**Activities:**
- Map control flow across the protocol.
- Identify all external entry points and their authorisation requirements.
- Catalogue fund-flow paths (deposits, withdrawals, mints, burns, transfers, fee accruals).
- Surface state-mutation invariants and where they are (or are not) enforced.
- Identify upgrade, governance, and pause surfaces.

**Output:** internal architecture map and invariant catalogue.

---

### 3. Manual Code Review

The core phase. Conducted by domain-experienced reviewers, not generated by tooling.

**Coverage:**
- Access control on every state-mutating entry point.
- Invariant enforcement on every fund-moving path.
- Mathematical correctness of accounting logic (rounding direction, overflow boundaries, decimal scaling).
- External call boundaries (reentrancy, untrusted callee, return-value handling).
- Signature verification paths (domain separation, replay protection, recovery validation).
- Oracle and price-feed assumptions.
- Upgrade and initialisation safety.
- Cross-contract trust assumptions.

Findings are recorded as they surface, with reproduction conditions documented.

---

### 4. Adversarial Analysis

Beyond bug-finding: how would a competent attacker approach this system?

**Activities:**
- Reverse the threat model — what attack does the protocol's incentive structure invite?
- Identify economic-attack vectors (sandwiching, oracle manipulation, governance capture, liquidity-flash exploits).
- Model failures under stressed market conditions (low liquidity, price gaps, validator/sequencer failures).
- Map cross-protocol composability risks if the protocol is exposed to integrations.

---

### 5. Finding Validation

Every candidate finding is:

- **Reproduced** — either via concrete transaction trace, foundry test, or step-by-step exploit walkthrough.
- **Classified** — severity assigned per the [severity rubric](SEVERITY.md), with impact and likelihood justified explicitly.
- **Documented** — title, severity, location, root cause, recommendation, fix plan.

Findings that cannot be reproduced are demoted to Informational or removed.

---

### 6. Reporting

Built from [`TEMPLATE.md`](TEMPLATE.md). Delivered in markdown and signed PDF.

**Contents:**
- Engagement summary.
- Severity-classified findings list.
- Per-finding entries with full root-cause analysis and remediation guidance.
- Out-of-scope acknowledgements.
- Methodology and tools appendix.

---

### 7. Remediation Review

After the client implements fixes:

- Each finding's fix commit is reviewed.
- Status updated: **Fixed** / **Acknowledged** / **Won't Fix** / **Mitigated**.
- "Fixed" requires the fix to address the **root cause**, not just the surface symptom.
- A finding can be re-opened if the fix introduces a new variant of the same class.

Remediation status is published in the final report.

---

## Communication Cadence

- Weekly written status update during active engagement.
- Critical and High findings are reported immediately on discovery, not at end of engagement, so the protocol team can begin mitigation work in parallel.
- Final report walkthrough with the protocol team before publication.

---

## What Refious Does Not Do

- We do not run automated scanners as a substitute for review and treat their output as findings.
- We do not file findings we cannot reproduce.
- We do not inflate severity counts.
- We do not perform engagements where the protocol team cannot or will not engage with the findings.
- We do not publish reports before disclosure windows close.

---

> A good audit closes the gap between what the protocol team thinks the system does and what an attacker would actually do with it.
